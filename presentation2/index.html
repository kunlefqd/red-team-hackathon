The Hackathon Challenge

The Mission Environment

  • Simulated quadcopter course (ProjectAirSim)
  • Unreal Engine environment
  • simple-flight autopilot · Python control API
  • Onboard FPV camera (640×480, 120° FOV)
  • Hard mode has no GPS or ground-truth
  • Final scene is held out and different

Why hard-coding fails

The final scene changes its colors, objects, counts, and geometry. A memorized path breaks instantly — the system has to generalize.

Feasibility

Feasible Because We're Not Starting From Scratch

  • simple-flight autopilot handles low-level stabilization
  • ProjectAirSim API issues control commands
  • OpenCV detects colors, arrows & sphere counts
  • Gemini VLM recognizes objects & returns pixel locations
  • Onboard state estimate navigates without GPS
  • Pydantic validates AI output; logging speeds iteration

The real work is integration

We're not inventing flight control from nothing — we're integrating proven tools into one working autonomy pipeline.

Technical Strategy

Our Build Plan

Phase 1 · Fly

  • Connect to control
  • Arm, take off, hover
  • Send stable setpoints

Phase 2 · See

  • Capture onboard FPV frames
  • OpenCV: colors, arrows, sphere counts
  • Gemini VLM: identify target vehicle

Phase 3 · Decide

  • Track mission state across stages
  • Remember both prior turns
  • Map turns → target vehicle (legend)

Phase 4 · Survive Hard Mode

  • Navigate by onboard state estimate (no GPS)
  • Steer with pixel→angle from the camera
  • Stop short / land beside on contact targets

Risks Become Design Requirements

Risk Our Response

Drone instability Lean on simple-flight autopilot; tune before speed
Debugging difficulty Log frames, masks & decisions to disk
