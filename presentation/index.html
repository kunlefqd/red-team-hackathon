The Problem · Entrepreneurial Drive

Uncrewed systems are losing their lifelines.

In modern CAF operations, uncrewed systems must increasingly operate in electronically contested and communication-denied environments — where remote control, data links, and GPS are jammed, spoofed, or simply unavailable.

  • GPS denied → no positioning oracle
  • Data link jammed → no human in the loop
  • Mission still has to be completed — autonomously

The gap

Most autonomy assumes a clean link and a GPS fix. Remove both and the platform is blind. The drone needs to perceive, reason, and decide on its own — turning raw camera pixels into navigation.

Our Solution

A drone that solves the puzzle, not just flies the route.

It navigates a branching course by reading visual clues at each junction — exactly the skill needed when no one can tell it where to go.

Perceive

Onboard camera frames are the only input. OpenCV + a vision-language model extract meaning.

Reason

Clues chain together — each decision narrows the objective until exactly one target remains.

Act

Pixel detections become yaw angles. The drone steers, approaches, and delivers — or stops short.

Algorithm Approach

The right tool for the right clue.

OpenCV — geometry & counting

  • HSV color masking isolates red/green arrows and blue spheres
  • Contour analysis reads arrow direction and sphere count
  • Fast, deterministic, runs every frame — no API cost

Gemini VLM — semantic recognition

  • Distinguishes a tank from a boat, jet, or ice-cream truck
  • Returns the target's pixel location on screen
  • Handles the open-ended "what am I looking at?" problem

Then the core logic: the two prior choices become the input to a decision table that outputs exactly one target vehicle.

Technical Implementation · Structured Vision

We don't parse prose — we get typed data.

Pydantic + Gemini's structured output force the model to return a validated schema, so the control loop receives fields it can act on directly.

class VLMResponse(BaseModel):
    target_visible: bool
    bbox_center_px: list[int]   # [x, y]
    distance_band:  DistanceBand   # far…very_close
    confidence:     float         # 0.0 – 1.0
  • target_visible — gate every decision on real detection
  • bbox_center_px — where the target is on screen
  • confidence — below 0.6 we hover & re-query instead of acting on a guess
Technical Implementation · Pixels → Motion

From a pixel on screen to a heading the drone can fly.

  • VLM gives the target's pixel (x, y)
  • Camera geometry (120° FOV) maps it to a yaw offset
  • Drone yaws onto the target, then closes distance
  • Re-query each tick — small errors self-correct, like a closed loop
def pixel_to_angles(x_px, y_px):
    dx = x_px - FRAME_W / 2
    yaw = (dx / (FRAME_W/2)) * (HFOV_DEG/2)
    # yaw > 0 → target is RIGHT, steer right
    return yaw_deg, pitch_deg
Technical Implementation · Safe Delivery

It knows the difference between a strike and a delivery.

Tank · Boat · Jet

Fly into the target to register the objective. Center on it, close the distance, confirm with the distance band.

Ice-Cream Truck

A delivery, not a collision. The drone stops short and lands beside it — using the distance_band to brake before impact.

The objective type changes the behavior — context-aware action, decided onboard.

Feasibility & Scalability

Riding the curve of frontier AI.

  • Feasible today and improving — every new premier model makes onboard reasoning cheaper, faster, and more reliable
  • Architecture is model-agnostic: swap in a better/on-device VLM as they arrive
  • The challenge: repeated API calls hit rate limits during testing

Scaling path

  • OpenCV does the high-frequency work; VLM is reserved for the hard semantic calls
  • Confidence gating + caching cuts redundant queries
  • Edge / on-device models remove the network dependency entirely
